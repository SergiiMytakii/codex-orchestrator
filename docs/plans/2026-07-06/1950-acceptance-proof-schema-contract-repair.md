## 1. Executive Summary
- **Goal:** Make Acceptance Proof reports easy for the Adaptive Proof Agent to write correctly and make invalid proof reports fail with complete, actionable schema feedback instead of one missing field per attempt.
- **Scope:** In scope: one source of truth for the Acceptance Proof report shape, generated prompt/template text, complete schema error collection, a local `acceptance-proof validate --report <path>` CLI check, a proof-agent draft/self-validation flow, and a proof-artifact repair path for malformed proof reports. Out of scope: changing GitHub publication authority, changing proof pass semantics, adding new browser/mobile tooling, running live smoke, or accepting product-code changes from proof.
- **Chosen Option:** Recommended delegated option: export a typed report contract from `src/runner/acceptance-proof.ts` and derive validation, prompt template text, and CLI validation from it.
- **Why This Approach:** It fixes the current drift without adding a new dependency or moving runner decisions into prompts. The Runner still owns pass/fail decisions; the prompt becomes a consumer of the same contract the Runner validates.

## 2. Current Understanding
- **Confirmed:** `src/runner/acceptance-proof.ts` defines `AcceptanceProofReport`, validates report shape in `assertAcceptanceProofReport`, and currently throws on the first schema violation. `readAcceptanceProofReport` converts that first exception into one invalid message. `src/runner/acceptance-proof-runner.ts` independently hardcodes a long prose `Schema: {...}` line in `buildAcceptanceProofPrompt`. `src/codex/command-adapter.ts` passes `reportPath` to Codex as the configured report file/`--output-last-message`, so a proof agent cannot validate the final saved report before exit unless the prompt explicitly tells it to write a draft or same-content report file first. `src/runner/acceptance-proof-loop.ts` turns invalid proof report reads into a blocked proof outcome. `src/runner/agent-attempt.ts` routes blocked publishability through `decideImplementationRework`, and `src/runner/rework-policy.ts` classifies any `Acceptance proof` reason as `failed-acceptance-proof`, which can schedule another implementation attempt. CLI help has `visual-proof ...` commands but no `acceptance-proof validate` command. `docs/adr/0002-adaptive-acceptance-proof.md` says the Proof Report must map each criterion to status, confidence, reasoning summary, and linked artifacts, while the Runner retains final authority.
- **Assumptions:** The intended fix should not add Ajv/Zod unless necessary; a narrow internal schema contract is enough because this package already uses hand-rolled validation. Invalid JSON/schema repair can be bounded to proof report artifact rewriting and proof-owned artifacts, while valid `needs-rework` proof continues to trigger implementation rework.
- **Open Decisions:** None blocking. The user has already specified the product direction: executable template, complete schema errors, proof-artifact repair for invalid schema, and local self-validation before proof exit.

## 3. Architectural Design
- **Component Flow:** Adaptive proof prompt builder asks `acceptance-proof.ts` for a minimal JSON template and validation command. The proof agent composes the report, writes the same JSON to `CODEX_ORCHESTRATOR_PROOF_REPORT_PATH` or a runner-declared draft path, runs `codex-orchestrator acceptance-proof validate --report <path>`, fixes any listed errors, then returns exactly the validated JSON as its final response so Codex `--output-last-message` overwrites the report with identical content. The Runner reads the report through the same `acceptance-proof.ts` schema validator, receives all schema errors at once, and classifies malformed reports as proof-report schema repair rather than implementation rework. Once the report is structurally valid, existing semantic evaluation still decides `passed`, `needs-rework`, or `blocked`.
- **Simplest Viable Path:** Keep `acceptance-proof.ts` as the deep domain module. Add exported helpers such as `validateAcceptanceProofReportShape`, `acceptanceProofReportTemplate`, and `formatAcceptanceProofSchemaErrors`. Replace the hardcoded prompt schema string with a generated fenced JSON template plus a validation command. Add a thin CLI command that reads a report and prints all shape errors. Update the prompt to explicitly allow proof report draft writes for self-validation while preserving the rule that product code and GitHub state are off limits. Add a bounded repair path in the acceptance proof loop that re-prompts the proof agent to repair only the proof report/artifacts when shape validation fails.
- **Why Not Simpler:** A prompt-only template would reduce first-pass failures but keep duplicate schema ownership and one-error validator behavior. A validator-only change would help rework but still leave the proof agent guessing. Adding Zod/Ajv would be heavier than necessary for this package unless later requirements need full JSON Schema interoperability.
- **Architecture Lens:** Module: `src/runner/acceptance-proof.ts` remains the Acceptance Proof report contract module. Interface: exported contract/template/validation helpers consumed by `acceptance-proof-runner.ts`, CLI, and loop tests. Seam: no new adapter abstraction is needed; adaptive proof and command proof already exist as concrete adapters behind `AcceptanceProofAdapterResult`. Deletion test: deleting the new contract helpers would immediately reintroduce prompt/validator drift, so the helper has real depth and leverage. Locality: prompt rendering, CLI validation, and loop classification all import from the one contract owner instead of restating field lists.
- **Clean Architecture Map:** Domain: `acceptance-proof.ts` owns report shape, shape validation, semantic evaluation, diff classification, and evidence builders. Application/Use Case: `acceptance-proof-loop.ts` owns proof attempt routing, invalid schema handling, proof repair loop bounds, and implementation rework decisions. Infrastructure: `acceptance-proof-runner.ts` owns Codex proof session execution and prompt file creation; `cli.ts` owns command parsing/output. Presentation: workflow prompt markdown and generated prompt text explain the contract but do not define it.
- **Reuse Strategy:** Reuse `AcceptanceProofReport`, `AcceptanceProofArtifact`, `AcceptanceProofCriterion`, `AcceptanceProofUiEvidence`, `readAcceptanceProofReport`, `buildBlockedAcceptanceProofOutcome`, `buildAcceptanceProofReportOutcome`, `runAcceptanceProofLoopAttempt`, `runAcceptanceProofAdapter`, and existing CLI parsing style in `src/cli.ts`. Reuse focused tests in `test/acceptance-proof.test.ts`, `test/acceptance-proof-loop.test.ts`, `test/scoped-auto-command.test.ts`, and `test/cli.test.ts`.
- **Rejected Paths:** Do not duplicate the schema in prompt text, tests, and validator. Do not route malformed proof JSON to a full scoped implementation attempt. Do not let the proof agent change product code to fix proof schema. Do not weaken semantic evaluation by treating a structurally valid report as proof success. Do not run `npm run smoke:live` for this change unless explicitly requested.

## 4. Constraints And Edge Cases
- **Data And Scale:** Proof reports are small JSON files; full in-memory parsing is acceptable. Error collection should cap path rendering to concise path strings but should not stop at the first array item or first missing field. Prompt templates must stay readable and avoid enormous generated schemas; the minimal template should cover required top-level and `criteria[]` fields, with optional `uiEvidence`, `proofScriptRepair`, and `reworkRequest` documented as generated optional sections or examples.
- **Errors And Fallbacks:** Invalid JSON should report parse failure plus no shape errors. Non-object reports, missing top-level fields, invalid enum values, malformed arrays, and malformed nested UI evidence should return all discoverable errors. Empty `criteria` remains a semantic failure after shape validation, because the shape allows an array but the proof standard requires at least one criterion. Missing artifact files remain semantic evaluation failures, not schema failures. CLI validation should exit `0` only for structurally valid shape and nonzero for parse/shape errors. If the proof agent validates a draft path, the final response must be byte-equivalent or JSON-equivalent to the validated draft; otherwise the runner's final read remains the source of truth and can still trigger repair/blocking.
- **Concurrency And State:** Proof report repair must be bounded by `reviewGates.acceptanceProof.maxIterations` or a smaller proof-repair budget, and must not increment full implementation attempts for pure schema failures. Repair prompts must use the same proof directory/report path and preserve proof-owned path restrictions. Repeated repair should be idempotent: overwrite the report, collect change set, then re-read through the same validator.

## 6. Execution Slices And Multi-Agent Model
- **Slices:** Slice 1: contract/template/error collector in `acceptance-proof.ts`. Slice 2: CLI validation command uses the same helper. Slice 3: prompt consumes generated template and gives the proof agent a draft/self-validation flow. Slice 4: acceptance proof loop treats invalid shape as bounded proof-artifact repair, not full implementation rework. Slice 5: documentation and final regression coverage.
- **Per-Slice Test/Proof:** Slice 1 starts with `test/acceptance-proof.test.ts` proving one malformed report returns multiple errors for missing `criteria[0].description`, `criteria[0].reasoningSummary`, `criteria[0].artifactRefs`, invalid status/confidence, missing `artifacts`, missing `proofPhaseDiff`, and missing `residualRisks` in one result. Slice 2 starts with `test/cli.test.ts` proving `acceptance-proof validate --report <path>` exits nonzero with all schema errors and exits zero for the existing passing fixture. Slice 3 starts with `test/scoped-auto-command.test.ts` proving the proof prompt contains a minimal valid JSON object with required fields, no hardcoded `Schema: { ... }` prose line, and explicit instructions to write the draft/same report JSON, run the validation command, then return the validated JSON as the final response. Slice 4 starts with `test/acceptance-proof-loop.test.ts` proving malformed proof report repair reuses the proof adapter/report path and does not schedule a full scoped implementation rework, while valid `needs-rework` still returns implementation rework evidence. Slice 5 runs focused tests plus final `npm run typecheck`, `npm test`, and `git diff --check`.
- **Exit Gates:** Each behavior-changing slice must have a failing test before implementation and pass the focused test after. Final gates: `npm run typecheck`, `npm test`, `git diff --check`. Do not run `npm run smoke:live` unless explicitly requested because it creates real GitHub side effects.
- **Agent Matrix:** Phase | Owner | Input | Output | Dependencies. Contract slice | main implementation agent | current `acceptance-proof.ts` interfaces/tests | exported shape validator/template | none. Prompt/CLI slice | main implementation agent | contract helpers | generated prompt and validation command | contract slice. Repair slice | main implementation agent | loop/rework policy | invalid-shape proof repair routing | prompt/CLI slice. Review slice | cleanup/code-review agents if implementation proceeds | final diff and tests | findings or approval | all implementation slices.
- **Parallelization Limits:** Do not implement prompt and validator in parallel before the contract helper exists. Do not write prompt self-validation instructions before the CLI command exists. Do not change rework policy before loop tests define invalid-shape versus valid-needs-rework behavior. Do not let a docs update become the only schema source.
