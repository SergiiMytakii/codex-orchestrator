## 1. Executive Summary
- **Goal:** Concentrate Acceptance Proof Loop decision and outcome handling behind one runner-facing module so adaptive proof sessions and runner-owned command proof use the same routing, diff, scope, report-evaluation, and evidence contract.
- **Scope:** In scope: one-attempt Acceptance Proof Loop orchestration, proof routing, adaptive-vs-command selection, proof phase diff capture, post-proof scope isolation inputs, Proof Report evaluation outcome assembly, Acceptance Proof attempt evidence, and `visualProof` migration adapter behavior. Out of scope: changing GitHub publication authority, running live smoke, changing the Proof Report JSON schema unless a test proves a missing field, adding UI proof tooling, implementing multi-iteration retry policy, or removing `reviewGates.visualProof` config in this change.
- **Chosen Option:** Add a sibling Acceptance Proof Loop module that orchestrates existing pure proof helpers and the two current adapters; this is the recommended option and preserves the user's requested centralized interface without turning `acceptance-proof.ts` into a god module.
- **Why This Approach:** `src/runner/acceptance-proof.ts` already has the right domain/evaluation primitives. A sibling `src/runner/acceptance-proof-loop.ts` is the smallest boundary that can depend on adapters, policy, git change-set collection, and scope isolation while keeping report schema/evaluation pure and reusable.

## 2. Current Understanding
- **Confirmed:** `CONTEXT.md` and ADR-0002 make Acceptance Proof runner-owned and require high-confidence Proof Reports with artifact mapping; `reviewGates.acceptanceProof` is canonical while `reviewGates.visualProof` is a migration adapter; `src/runner/acceptance-proof.ts` already owns report evaluation and diff classification; `acceptance-proof-runner.ts`, `visual-proof-runner.ts`, `local-execution-session.ts`, and `review-gate-policy.ts` still independently choose or interpret proof behavior; tests already cover report evaluation and visual command evidence.
- **Assumptions:** Existing CLI/config compatibility for `codex-orchestrator visual-proof ...` must remain; command proof and adaptive proof are the two real adapters for the same Acceptance Proof Loop Interface; focused local tests can prove this refactor without live smoke unless the user explicitly requests it; `beforeHead` is captured by the local execution session before child implementation and remains the base for post-proof full change-set collection.
- **Open Decisions:** None before planning. User approval is still required before implementation.

## 3. Architectural Design
- **Component Flow:** `local-execution-session.ts` passes issue, config, initial changed files, completion report, adapter callbacks, `beforeHead`, git change-set collection, and a scope-isolation evaluator into `acceptance-proof-loop.ts`. The loop module resolves proof routing, creates one proof baseline before adapter execution, invokes either the Adaptive Proof Agent adapter or runner-owned command proof adapter, recollects the full final change set after proof using `beforeHead`, derives the proof-phase diff, evaluates the Proof Report through `acceptance-proof.ts`, evaluates scope isolation on the full final changed paths, and returns one runner-facing outcome. Local execution only merges artifacts/risks and blocks or proceeds from that outcome.
- **Simplest Viable Path:** Create `src/runner/acceptance-proof-loop.ts` with a small public API such as `planAcceptanceProofAttempt` and `runAcceptanceProofLoopAttempt`. Keep `src/runner/acceptance-proof.ts` as pure domain/evaluation/builders; it must not import `acceptance-proof-runner.ts`, `visual-proof-runner.ts`, `review-gate-policy.ts`, git, shell, or local-session code.
- **Why Not Simpler:** Moving only routing or only outcome builders would still leave proof blockers scattered across local session, visual command proof, and adaptive proof. Putting orchestration into `acceptance-proof.ts` would reduce file count but would invert dependencies by mixing pure report evaluation with adapter invocation.
- **Architecture Lens:** Module: Acceptance Proof Loop (`acceptance-proof-loop.ts`) plus the existing Acceptance Proof domain helpers (`acceptance-proof.ts`). Interface: a runner-facing attempt API returning `AcceptanceProofAttemptEvidence`, final `changedFiles`, validation lines, artifacts, residual risks, blockers, and first-class `scopeBlockers`. Seam: two real adapters exist now, Adaptive Proof Agent and runner-owned command proof, so a minimal adapter result contract is warranted. Deletion test: deleting the loop module would push routing, diff capture, report evaluation, forbidden diff handling, and post-proof scope isolation back into multiple callers. Depth/Leverage/Locality: the loop hides proof decision complexity behind one call while keeping external execution and publication boundaries explicit.
- **Clean Architecture Map:** Domain: proof statuses, criteria, artifacts, UI Evidence, proof diff classification, Proof Report evaluation in `acceptance-proof.ts`. Application/Use Case: proof routing, one-attempt loop orchestration, adapter result normalization, diff/scope outcome assembly in `acceptance-proof-loop.ts`. Infrastructure: Codex adapter, shell executor, git change-set collection, filesystem artifact checks, screenshot discovery inside adapters. Presentation: prompts, validation summaries, CLI command strings, final handoff evidence.
- **Reuse Strategy:** Reuse `AcceptanceProofReport`, `AcceptanceProofAttemptEvidence`, `evaluateAcceptanceProofReport`, `createAcceptanceProofDiffCapture`, `classifyAcceptanceProofDiff`, `buildAcceptanceProofReportOutcome`, `buildBlockedAcceptanceProofOutcome`, `buildForbiddenAcceptanceProofDiffEvidence`, existing routing policy inputs, and existing tests. Add a tiny raw adapter result type with status, report path, artifact paths, stdout/stderr or session summary, exit code/error, and residual risks; final evidence construction belongs only in the loop. `review-gate-policy.ts` should either export deprecated wrappers delegating to the new loop planning API or retain only prompt text helpers.
- **Rejected Paths:** Do not make `acceptance-proof.ts` import adapters or policy orchestration. Do not let proof agents mutate labels, comments, PRs, branches, or publication state. Do not make visual proof canonical. Do not duplicate Proof Report evaluation in adapters. Do not allow legacy `visualProof` policy to expand proof-owned mutation paths; allowed proof repair paths come from `reviewGates.acceptanceProof.proofOwnedPathGlobs`. Do not remove legacy `visualProof` config before compatibility tests prove delegation. Do not introduce an import cycle where `visual-proof-runner.ts` imports the loop while the loop imports `visual-proof-runner.ts`; keep raw adapter functions separate from legacy wrapper entrypoints if compatibility requires both.

## 4. Constraints And Edge Cases
- **Data And Scale:** Proof reports and artifact lists are small per issue, but changed-file lists may be large; keep normalized sorted arrays and set-based membership as current helpers do. Avoid additional artifact content reads beyond existing hash snapshots needed by command proof. No pagination or N+1 concerns.
- **Errors And Fallbacks:** Missing/invalid Proof Report blocks with `AcceptanceProofAttemptEvidence`; command nonzero plus passing report blocks; screenshot-only output remains insufficient; missing artifact paths fail evaluation; unsupported proof strategy returns explicit skip/error according to policy; missing browser/device tooling remains concrete skipped proof evidence only where current policy allows non-blocking behavior.
- **Concurrency And State:** The loop module is the single owner of before/after proof diff capture and final changed-file recollection. Inputs must include both `initialChangedFiles` and `beforeHead`; the loop must create the proof baseline itself before adapter execution, must not trust adapter-provided changed files for forbidden diff classification, and after adapter execution must call git collection once, compute `proofPhaseChangedFiles`, classify forbidden product paths, and run post-proof scope isolation against the full final changed paths. Proof-owned repairs are only paths matching `reviewGates.acceptanceProof.proofOwnedPathGlobs`; product-code proof diffs block and become rework evidence. Multi-iteration retry remains outside this one-attempt refactor and must continue to respect `maxIterations` when later wired.

## 5. Impacted Areas
- `src/runner/acceptance-proof-loop.ts`: new application module owning proof routing, adapter choice, diff capture, post-proof scope input handling, and final evidence/outcome assembly.
- `src/runner/acceptance-proof.ts`: remains domain/evaluation helper module; may receive pure helper extractions only, not adapter imports.
- `src/runner/acceptance-proof-runner.ts`: becomes the Adaptive Proof Agent adapter; keeps prompt construction, Codex execution, isolated home, and report path setup; returns raw adapter result for the loop.
- `src/runner/visual-proof-runner.ts`: becomes the runner-owned command proof adapter; keeps shell execution, env setup, screenshot freshness detection, and legacy CLI command behavior; final evidence building moves to the loop.
- `src/runner/local-execution-session.ts`: loses adaptive-vs-command branching, runner proof diff classification, and post-proof scope blocking details; consumes a single proof loop outcome.
- `src/runner/review-gate-policy.ts`: keeps prompt/policy rendering; `decideProofRouting` and `shouldApplyVisualProofGate` either move to the loop planning API or become thin delegating wrappers with a regression test proving no independent strategy choice remains.
- Tests: add direct loop-interface coverage and preserve current public behavior in `test/acceptance-proof.test.ts`, `test/local-execution-session.test.ts`, `test/visual-proof-runner.test.ts`, and routing-policy tests if present.
