## 1. Executive Summary
- **Goal:** Make `agent:plan-auto` recover and continue safe, runner-owned parent-tree work after stale branches, closed successful children, and retryable child blockers, instead of immediately re-blocking the parent with manual cleanup instructions.
- **Scope:** In scope: deterministic recovery decisions for `plan-parent` and `tree-child` worktrees, reuse of durable run summaries, resume of already merged child work, bounded rework for retryable child blockers, tests, docs, and one focused live-smoke scenario. Out of scope: force-deleting unmerged work, changing standalone scoped recovery semantics except shared helpers, merging PRs, or adding a generic workflow engine.
- **Chosen Option:** Option 2, delegated recommended option: add a deterministic plan-auto tree recovery decision that resumes safe state and hard-blocks unsafe state.
- **Why This Approach:** The current runner already owns worktrees, local state, durable summaries, labels, publication, and rework decisions. The smallest sufficient fix is to teach that same runner to classify existing plan-auto tree state before worktree creation, then either resume from proven runner-owned artifacts or block with exact recovery evidence. This avoids both manual runbooks and unsafe branch resets.

## 2. Current Understanding
- **Confirmed:** `CONTEXT.md` defines the Runner as the trusted owner of issue selection, worktree setup, validation, GitHub mutations, publication, durable state, and bounded rework. `docs/adr/0001-runner-owned-loop-policy.md` explicitly keeps loop policy, rework bounds, durable memory, and publication runner-owned. `src/git/worktree.ts` already has `ensureIssueWorktree(... allowResume)` for resumable worktrees, while `runPlanAutoCommand()` and `executeChild()` currently call strict `createIssueWorktree()` before parent planning and child execution. `src/runner/scoped-recovery.ts` contains stale-run classification and bounded scoped recovery for standalone issues, but not parent-tree recovery. `src/runner/issue-tree.ts` currently treats every closed child as an executable graph error. `test/plan-auto-command.test.ts` already covers child rework before parent integration. `scripts/live-smoke.mjs` already has `tree-child-quality-rework`, proving the normal rework path but not stale parent/tree resume.
- **Assumptions:** The target failure shape is represented by a runner-owned parent branch such as `codex/tree-262`, one or more child branches/worktrees such as `codex/tree-262-issue-265`, durable state under `runner.stateDir`, and GitHub child issues carrying the configured child label plus parent marker. Existing local summaries are trusted only when the issue number, parent issue number, branch name, worktree path, and mode match the current target configuration.
- **Open Decisions:** None for planning. Implementation must preserve hard-block behavior for dirty, foreign, ambiguous, or unprovable state.

## 3. Architectural Design
- **Component Flow:** `runPlanAutoCommand()` resolves config/base/parent issue, then calls a new plan-auto recovery classifier before creating parent or child worktrees. The classifier reads local runner state, worktree metadata, Git branch ancestry, current GitHub issue state, and durable summaries. It returns a typed decision: start fresh, resume parent, treat a child as already completed, resume child rework, or hard-block. The existing parent execution loop consumes the decision, skips already completed children, reworks retryable blocked children through the existing `decideImplementationRework()` path, then continues normal merge, final validation, PR handoff, and label updates.
- **Simplest Viable Path:** Add a focused `src/runner/plan-auto-recovery.ts` module with pure decision functions plus narrow IO helpers. Wire it into `src/runner/plan-auto-command.ts` at two points only: before parent worktree creation and before each child execution. Reuse `RunnerStateStore`, `RunnerLifecycleEventStore`, `GitWorktreeManager.ensureIssueWorktree`, durable summary readers, and `decideImplementationRework`.
- **Why Not Simpler:** Replacing `createIssueWorktree()` with `ensureIssueWorktree()` alone is not enough. A clean parent branch can contain merged child commits, closed successful children currently fail `collectExecutableChildBatches()`, and a child branch can have retryable blocked work that should enter rework rather than fresh execution. The added plan-auto recovery module is required to centralize those state decisions and prevent scattered branch/label heuristics.
- **Architecture Lens:** Module: `plan-auto` execution. Interface: a typed `PlanAutoRecoveryDecision` consumed by `runPlanAutoCommand()` and `executeChild()`. Seam: runner-owned recovery classification over existing git/GitHub/local-state adapters. Deletion test: deleting this module would push recovery rules back into `plan-auto-command.ts`, `worktree.ts`, and issue-tree scheduling, duplicating safety checks and making branch deletion/resume rules harder to audit. Depth/Leverage/Locality: one deep recovery policy concentrates a high-risk state machine near the current parent-tree owner without inventing a new adapter layer.
- **Clean Architecture Map:** Domain: recovery decision types, retryability rules, and safety invariants. Application/Use Case: `runPlanAutoCommand()` orchestration and child scheduling. Infrastructure: Git worktree inspection, local JSON state, GitHub issue reads/writes, durable summary file reads. Presentation: CLI `status`/daemon output and GitHub comments remain runner-owned reports, not agent-authored decisions.
- **Reuse Strategy:** Reuse `RunnerStateStore` from `src/runner/local-state.ts`, lifecycle events from `src/runner/lifecycle-events.ts`, worktree inspection/resume from `src/git/worktree.ts`, rework decisions from `src/runner/rework-policy.ts`, durable summaries from `src/runner/durable-run-summary.ts`, plan graph parsing from `src/runner/issue-tree.ts`, and existing plan-auto reporting helpers in `src/runner/plan-auto-command.ts`.
- **Rejected Paths:** Do not force-delete unmerged branches. Do not silently reopen or close GitHub issues without runner-owned evidence. Do not let the Codex Agent decide whether stale work is safe. Do not build a generic workflow-resume engine beyond `plan-auto`. Do not treat all closed child issues as success; only runner-owned review-ready summaries and matching branches can be skipped as completed.

## 4. Constraints And Edge Cases
- **Data And Scale:** Recovery reads one parent run, the parent graph, child issue metadata, matching runner-state entries, and bounded durable summaries for children in that graph. It must not scan every historical summary file in large repositories; lookup should be by current runner metadata first and by deterministic session prefix only as a fallback.
- **Errors And Fallbacks:** Missing GitHub issue, missing parent marker, invalid summary JSON, dirty parent worktree, dirty child worktree without retryable blocker, branch created from a different base, missing required base evidence, foreign hostname/lease where applicable, and mismatch between summary parent/child numbers all hard-block. GitHub or filesystem read failures should produce explicit blocked reasons, not cleanup.
- **Concurrency And State:** Daemon plan-auto remains exclusive. Recovery must be idempotent: repeated daemon passes over the same stale parent tree produce the same decision and do not duplicate child comments, PRs, merges, or issue labels. Running same-host processes remain active, not recoverable. Closed successful children can be skipped only once and included in final parent reporting as recovered child results. Retry attempts remain bounded by `loopPolicy.rework.maxAttempts` and `acceptanceProof.maxIterations - 1`.

## 6. Execution Slices And Multi-Agent Model
- **Slices:**
  1. Parent recovery decision tracer bullet: write failing tests for an existing clean `codex/tree-<parent>` worktree that belongs to the current parent and should be resumed, plus dirty/foreign variants that hard-block. Implement `plan-auto-recovery` parent classification and wire parent worktree creation through `ensureIssueWorktree` only when the decision allows it.
  2. Completed child recovery: write failing tests where a child issue is closed/review-ready with a matching durable summary and child branch already merged into the parent branch. Implement recovered child result reconstruction and scheduling so this child is skipped as completed and appears in parent handoff evidence.
  3. Retryable child rework resume: write failing tests where an existing `tree-<parent>-issue-<child>` worktree has prior blocked summary for a retryable quality-gate reason. Implement child recovery so it enters the existing bounded rework loop with a rework prompt instead of creating a new worktree or blocking on branch existence.
  4. Unsafe-state reporting: write failing tests for dirty parent worktree, branch from wrong base, closed child without matching summary, blocked child with non-retryable reason, missing report, and already alive runner process. Implement stable blocked comments/events without branch deletion.
  5. Status/daemon/live-smoke coverage: extend status/recovery reporting where needed, add a `plan-auto-tree-recovery` live smoke scenario using the fake agent, and add script tests proving the scenario/profile wiring.
  6. Documentation and final validation: update docs, run full local validation, then run the focused live smoke only when explicitly approved because it mutates the scratch GitHub repository.
- **Per-Slice Test/Proof:** Slice 1: Node tests assert `runPlanAutoCommand()` resumes a clean parent worktree and does not add `agent:blocked`. Slice 2: tests assert closed successful child is not re-executed and final PR/report includes recovered child evidence. Slice 3: tests assert existing child worktree receives `automatic rework attempt (#1)` and succeeds through existing quality gate. Slice 4: tests assert each unsafe state blocks with a concrete reason and preserves worktrees. Slice 5: `test/live-smoke-script.test.ts` asserts scenario registration and fake-agent markers; live proof command is `npm run smoke:live -- --scenario plan-auto-tree-recovery --cleanup`. No UI proof is needed because this is CLI/runner behavior.
- **Exit Gates:** Per slice: `npm run typecheck` when types or module boundaries change; targeted `npm test -- test/plan-auto-command.test.ts` after plan-auto changes. Final local gate: `npm run typecheck`, `npm test`, `git diff --check`. Live gate when approved: `npm run smoke:live -- --scenario plan-auto-tree-recovery --cleanup`.
- **Agent Matrix:** Phase | Owner | Input | Output | Dependencies
  - Recovery policy tests | Single implementation agent | current plan-auto/worktree/local-state behavior | failing then passing unit tests for decisions | none
  - Parent/child wiring | Single implementation agent | recovery decision API | resumed parent and child flows | policy tests
  - Live smoke | Single implementation agent | packaged CLI/fake agent harness | focused scenario proof | local tests green
  - Review/signoff | Same agent plus normal cleanup/code review gates | final diff and validation evidence | review-ready implementation | all prior phases
- **Parallelization Limits:** Do not implement parent recovery and child recovery in parallel because they share `runPlanAutoCommand()` state and final reporting. Do not run live smoke while local behavior tests are red. Do not run multiple daemon/live smoke scenarios against the same scratch repo run id concurrently.
