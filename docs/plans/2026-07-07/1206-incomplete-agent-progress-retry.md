## 1. Executive Summary
- **Goal:** Let scoped issue execution continue when an Agent idle-times out after making runner-verifiable safe local progress, so blocking happens only when state is unsafe, unprovable, or the bounded rework budget is exhausted.
- **Scope:** Add a narrow retryable blocker for exact idle-timeout-after-progress in scoped issue publishability/rework policy, update config schema/defaults/repo config, tests, and docs. Out of scope: plan-auto/tree recovery, blanket timeout retry, increasing Codex timeout as the fix, live smoke execution, resuming arbitrary stale dirty worktrees, or allowing Agent-owned publication.
- **Chosen Option:** User selected the structural recommended option: runner-classified bounded retry after safe local progress.
- **Why This Approach:** It preserves the current hard boundary that unknown Codex exits block, while converting one machine-checkable subset into normal bounded rework: exact idle timeout, missing completion report, non-empty safe diff, no publication violation, no denied paths, scope isolation pass, and remaining retry budget.

## 2. Current Understanding
- **Confirmed:** `src/process/command.ts` returns `exitCode: 124` and stderr line `Command idle timed out after <ms>ms.` for idle timeouts. `.codex-orchestrator/config.json` sets `codex.idleTimeoutMs` to `300000`. `src/runner/local-execution-session.ts` currently blocks all nonzero Codex exits before reading reports or collecting diff. `src/runner/rework-policy.ts` treats unmatched reasons as `unknown`, and `unknown` is hard-block. Existing config retries only listed blocker keys. Real #1210 had a two-file safe diff but no completion report, and the durable summary showed `changedFiles: []` because publishability never collected diff.
- **Assumptions:** The correct continuation is the existing `agent-attempt` rework loop in the same worktree, not a separate recovery subsystem. The initial repo-local and default setup policy should include the new blocker with the existing `maxAttempts` budget, because the behavior should be available to future configured repos rather than only this checkout.
- **Open Decisions:** None before implementation. If maintainers later want different budgets for incomplete-progress versus other rework blockers, that is a separate config design.

## 3. Architectural Design
- **Component Flow:** Agent attempt runs -> process executor returns idle-timeout result -> publishability first checks agent-owned publication violations -> exact idle-timeout branch inspects report state and collects the change set -> valid reports continue through normal report-based publishability; missing reports evaluate safety gates that do not require a report -> safe non-empty progress becomes sentinel blocker `INCOMPLETE_AFTER_PROGRESS_REASON` with collected `changedFiles` -> `decideImplementationRework()` maps sentinel to `incomplete-after-progress` -> existing rework loop starts the next Agent attempt in the same worktree -> next attempt must finish the existing diff, run validation, and emit the normal completion JSON -> ordinary publishability, configured checks, Acceptance Proof, durable summary, and handoff continue.
- **Simplest Viable Path:** Add one new rework blocker key, one exported sentinel reason, and one publishability branch before the generic nonzero-exit block. The branch must match only `exitCode === 124` plus exact idle-timeout stderr text (`Command idle timed out after <positive integer>ms.`). Generic `Command timed out after ...`, arbitrary `exitCode 124`, and nonzero exits remain `unknown` hard-blocks.
- **Why Not Simpler:** A blanket `code 124` retry would retry unknown or globally timed-out attempts without proving progress. Raising `idleTimeoutMs` would only delay the same failure and would still lose recoverable local work when the Agent goes silent before final JSON.
- **Architecture Lens:** Module: scoped Agent attempt publishability. Interface: `runImplementationPublishabilityCheck()` returns blocked/publish-ready/promotion with changedFiles/evidence; `decideImplementationRework()` decides retry/exhausted/hard-block from runner-owned reason strings. Seam: no new subsystem; a tiny local helper such as `isIdleTimeoutResult()` is acceptable inside `local-execution-session.ts` because it concentrates an exact process-result predicate. Deletion test: deleting the helper should not remove a layer, only inline the predicate, so no new module is justified. Depth/Leverage/Locality: classification stays near change-set and safety evidence, while retry budgeting stays in rework policy.
- **Clean Architecture Map:** Domain policy: `src/runner/rework-policy.ts` owns blocker keys, sentinel reason, retryability, and hard-block taxonomy. Application/use case: `src/runner/local-execution-session.ts` owns publishability classification from process result, report state, diff, and safety gates; `src/runner/agent-attempt.ts` owns retry execution and prompt context. Infrastructure: `src/process/command.ts` remains the source of idle-timeout output but should not gain runner policy. Presentation: `src/runner/durable-run-summary.ts` and `src/runner/handoff-evidence.ts` render existing changedFiles/blocker evidence.
- **Reuse Strategy:** Reuse `git.collectSessionChangeSet()`, `validateNoAgentOwnedGitPublication()`, `validateChangedPaths()`, `evaluateScopeIsolation()`, `readScopedCompletionReport()`, `decideImplementationRework()`, existing rework prompt plumbing, durable summary writing, and blocked handoff rendering. Do not duplicate safety matching in rework policy.
- **Rejected Paths:** Blanket retry for all `exitCode 124`; parsing timeout text inside `rework-policy` as the source of truth; accepting dirty worktree without completion report as publish-ready; retrying unsafe denied paths or publication violations; retrying arbitrary stale dirty worktrees; adding a new recovery daemon/scanner; changing live smoke; letting the Agent write GitHub labels/comments/PRs.

## 4. Constraints And Edge Cases
- **Data And Scale:** Changed file lists are already collected for publishability and are bounded by repo diff size; no pagination or external data fetching. Do not read full file contents for classification.
- **Errors And Fallbacks:** Safe incomplete progress requires all of: exact idle-timeout predicate, no agent-owned Git publication violation, missing report, non-empty changed paths, `validateChangedPaths()` has no violations, and `evaluateScopeIsolation()` has no blockers. If the report is valid, run normal report-based publishability for that attempt instead of classifying it as incomplete progress. If the report is invalid/unreadable, keep the existing invalid-completion-report path; do not collapse invalid reports into incomplete progress in the first slice. Without a report, do not infer prohibited or destructive actions by scraping the transcript; use only existing runner-owned evidence available without a completion report: publication violation checks, changed path safety, and scope isolation. If no diff exists, use existing no-progress/missing-report behavior. If denied paths, scope isolation, publication violation, generic timeout, arbitrary `exitCode 124`, or non-idle nonzero exit appears, hard-block with the existing reason taxonomy.
- **Concurrency And State:** The retry must reuse the existing attempt loop and same worktree, increment attempt/rework evidence, preserve prompt/log/report/snapshot paths per attempt, and keep terminal GitHub mutations idempotent. On exhausted retry budget, terminal blocked durable summary/comment must include the collected `changedFiles` so #1210-style `changedFiles: []` does not repeat.

## 5. Impacted Areas
- `src/runner/local-execution-session.ts` — add exact idle-timeout-after-progress branch before generic nonzero exit and return collected `changedFiles` on retryable/exhausted blocked evidence.
- `src/runner/rework-policy.ts` — add `INCOMPLETE_AFTER_PROGRESS_REASON`, `incomplete-after-progress` blocker key, classifier pattern for the sentinel only, and retryability support without parsing raw timeout text.
- `src/config/schema.ts`, `.codex-orchestrator/config.json`, setup/default config fixtures/tests — allow and configure the new blocker key for this repo and generated defaults.
- `src/runner/agent-attempt.ts` / `src/runner/prompt.ts` only if current rework prompt does not include enough changed-file context; otherwise leave unchanged and rely on blocker reason plus changedFiles evidence.
- `test/rework-policy.test.ts`, `test/scoped-auto-command.test.ts`, `test/local-execution-session.test.ts` or equivalent public behavior tests — cover policy and end-to-end scoped behavior.
- `docs/deep-dive.md` — document the narrow retryable state and hard-block boundary.

## 6. Execution Slices And Multi-Agent Model
- **Slices:** 1. Safe-progress behavior: add a scoped/public behavior test where first Codex run exits with exact idle timeout after writing allowed files and no report; runner reworks; second run emits valid report; result becomes review-ready and rework prompt mentions previous blocker and changed files. 3. Publishability implementation: implement the exact idle-timeout branch in `local-execution-session.ts` using existing change-set and safety gates, preserving normal paths for valid report, missing/no-diff, invalid report, and generic nonzero exits. 4. Hard-block edges: add tests for denied path, scope isolation blocker, publication violation, generic command timeout, arbitrary `exitCode 124`, exhausted rework budget, and terminal summary/comment retaining changedFiles. 5. Documentation and config defaults: update `.codex-orchestrator/config.json`, setup/default config validation expectations, and `docs/deep-dive.md`.
- **Per-Slice Test/Proof:** Each behavior-changing slice starts with a failing Node test and ends green. Final proof: focused tests touched above, `npm run typecheck`, `npm test`, and `git diff --check`. No UI proof applies.
- **Exit Gates:** A final cleanup-review and code-review gate should run before implementation handoff because runner publication policy changes are medium-risk. Live smoke is skipped unless explicitly requested, per repo policy.
- **Agent Matrix:** Phase | Owner | Input | Output | Dependencies. Plan review | subagent | draft plan and evidence | review verdict | none. Implementation | main/downstream implementer | approved plan | code/docs/tests | plan approval. Final review | main/downstream implementer | diff and validation | review findings fixed or documented | implementation complete.
- **Parallelization Limits:** Do not split implementation across agents unless write scopes are disjoint. `local-execution-session`, `rework-policy`, config schema, and scoped tests are coupled and should be implemented serially.
