## 1. Executive Summary
- **Goal:** Add a local-only daily self-improvement loop for this repository: discover one architecture improvement, create an `agent:auto` issue for it, run codex-orchestrator on that exact issue so it can open a draft PR, run live smoke after the issue implementation phase, then review completed self-improvement work and create manual follow-up issues for process problems.
- **Scope:** In scope: ignored local runner files under `.codex-orchestrator/local/self-improvement/`, Codex App automation, GitHub issue creation, one architecture issue per daily run, targeted issue implementation, post-implementation `npm run smoke:live`, daily review follow-up issue creation, and removal of accidental package WIP. Out of scope: adding this feature to the npm package, changing `src/` runner behavior, changing package docs/release notes, auto-merge, or auto-release.
- **Chosen Option:** Option 2, Local Ignored Runner + Codex App Automation, with live issue creation and daily discovery plus daily review.
- **Why This Approach:** It gives the self-improvement loop enough local state and idempotency to be repeatable, while keeping the published package a controlled GitHub Issues runner rather than a self-managing product.

## 2. Current Understanding
- **Confirmed:** `CONTEXT.md` defines Codex Orchestrator as a controlled issue runner, with the Runner owning issue selection, GitHub mutation, validation, and publication handoff. ADR-0001 requires loop policy to remain runner-owned and deterministic. `docs/deep-dive.md` confirms `agent:auto` authorizes scoped implementation, `agent:manual` reserves an issue for human work, targeted `run --issue` execution can open draft PRs for one eligible issue, and the runner never auto-merges. `.codex-orchestrator/config.json` already points at `SergiiMytakii/codex-orchestrator` and includes the required labels. `.gitignore` already ignores runtime state under `.codex-orchestrator/state/`, workspaces, and proofs, but not `.codex-orchestrator/local/`.

A `self-improvement` GitHub label either exists or can be created by the local runner. Codex App automation can run a recurring local prompt against this workspace.
- **Open Decisions:** None blocking before implementation. During implementation, if GitHub refuses label creation, the runner should fail loudly instead of creating unlabeled self-improvement work.

## 3. Architectural Design
- **Component Flow:** Codex App automation wakes daily, runs the local self-improvement runner, the runner executes shared preflight, then runs discovery, then runs implementation for the exact discovery issue with `codex-orchestrator run --target . --issue <created-or-reused-issue>`, then runs `npm run smoke:live` after the implementation phase, then runs review even if discovery, implementation, or live smoke failed. Review scans prior self-improvement work ready for review and creates `agent:manual` follow-up issues for workflow problems.
- **Simplest Viable Path:** Use one local Node script or small local command tree under `.codex-orchestrator/local/self-improvement/`. Store minimal state in the same ignored directory: last run timestamp, created discovery issue numbers, reviewed source fingerprints, and created follow-up issue numbers. Use `gh` for labels/issues and `codex-orchestrator run --target . --issue <number>` for deterministic implementation of the newly created or reused discovery issue.
- **Why Not Simpler:** A prompt-only automation would be faster, but it would duplicate idempotency and label rules inside an untested prompt. A tiny local runner is justified because duplicate issue prevention, review state, and error reporting need a single local source of truth.
- **Architecture Lens:** The local self-improvement runner is a local Module with a command-line Interface: `daily`, `discover`, `implement`, and `review`. The Seam is process execution of `gh`, Codex skill invocation, and `codex-orchestrator run`; each starts as a concrete local command call rather than an Adapter abstraction. The deletion test passes: deleting the local runner would scatter idempotency, label policy, target issue selection, and review bookkeeping across Codex automation prompts. The Module increases Depth by hiding daily-loop bookkeeping behind a small local command, and improves Locality by keeping all private self-improvement rules in one ignored directory.
- **Clean Architecture Map:** Domain: self-improvement candidate, review finding, local run state, created issue record. Application/Use Case: daily orchestration, discovery issue creation, daemon invocation, review follow-up creation, idempotency checks. Infrastructure: `gh`, filesystem state, Codex App automation, `codex-orchestrator` CLI. Presentation: CLI output and Codex App automation summary.
- **Reuse Strategy:** Reuse `.codex-orchestrator/config.json` for GitHub owner/repo and canonical agent labels. Reuse `codex-orchestrator run --target . --issue <number>` for targeted implementation and draft PR handoff. Reuse the `improve-codebase-architecture` skill for discovery language, but wrap it in a local JSON output contract so the runner can validate and fingerprint candidates before creating an `agent:auto` issue. Reuse GitHub Issues as the Issue Work Queue rather than adding a separate queue.
- **Rejected Paths:** Do not add commands, config schema, runner modules, tests, README entries, or package exports for this local loop. Do not put review follow-ups on `agent:auto`. Do not auto-merge, publish, or run `npm publish`. Do not let the discovery prompt create more than one implementation issue per daily run. Do not run live smoke before the targeted implementation phase completes or outside the local self-improvement runner's explicit daily flow.

## 4. Constraints And Edge Cases
- **Data And Scale:** Daily discovery handles a small candidate list and creates only the first validated candidate after the discovery prompt's stable ordering. Review scans only self-improvement issues created by the local runner and caps each daily review batch to five source issues and five follow-up findings per source issue.
- **Errors And Fallbacks:** Shared preflight failures stop all phases: wrong GitHub repo, missing `gh` auth, missing `agent:auto` or `agent:manual`, uncreatable `self-improvement` label, missing Codex command, or an existing active self-improvement lock. After shared preflight passes, `daily` records separate phase results and always attempts review even if discovery, targeted implementation, or live smoke fails. If `improve-codebase-architecture` produces no valid candidate, no discovery issue is created and the run records a no-op. If issue creation succeeds but local state write fails, the next run recovers from remote idempotency markers before creating anything. If targeted `run --issue` fails, live smoke is skipped for that issue with a recorded reason, and review still runs over previously eligible completed work. If `npm run smoke:live` fails, the runner records the smoke failure in local state and includes it in the daily summary; it does not auto-close, auto-merge, or relabel issues beyond the existing codex-orchestrator runner behavior.
- **Concurrency And State:** Use atomic lock acquisition with `mkdir .codex-orchestrator/local/self-improvement/lock`; record owner pid, hostname, and timestamp; allow stale lock recovery only when the recorded process is absent and the lock age exceeds the configured timeout. Before any GitHub mutation, search GitHub by remote idempotency markers, not local state alone. Every discovery issue body includes `self-improvement-runner-id`, `source-candidate-fingerprint`, and `source-date`. Every review follow-up includes `self-improvement-runner-id`, `source-issue`, `source-pr`, and `finding-fingerprint`.

### Discovery Contract
- The local runner invokes Codex with the `improve-codebase-architecture` skill and requires raw JSON output:

```json
{
  "status": "completed",
  "candidates": [
    {
      "title": "short imperative title",
      "files": ["repo-relative/path.ts"],
      "problem": "why the current architecture causes friction",
      "solution": "plain-language implementation direction",
      "benefits": ["locality/leverage/testability benefit"],
      "verification": ["observable test or command"],
      "risk": "main risk or none",
      "adrConflict": "none or explicit ADR reference"
    }
  ],
  "residualRisks": ["string"]
}
```

- The prompt must order candidates by: smallest blast radius first, then clearest verification, then no ADR conflict, then lexical title. The runner takes `candidates[0]` only after validating required fields.
- The runner rejects candidates with empty files, empty verification, an ADR conflict, or no concrete implementation direction. Rejected discovery output creates no GitHub issue and records the rejection.
- The discovery issue title is `Self-improvement: <candidate.title>`. The body contains the candidate fields, explicit acceptance criteria, `## codex-orchestrator metadata` ownership bullets for candidate files, and remote markers. Labels are exactly `agent:auto` and `self-improvement`.

### Review Contract
- A source issue is review-eligible only when it was created by this runner, has the `self-improvement` label, has the runner marker, and is in `agent:review`, closed with linked PR evidence, or has a linked draft/open PR created by codex-orchestrator. Issues still carrying `agent:running` or `agent:blocked` are skipped.
- Review evidence is limited to the source issue body, comments, labels, linked PR metadata/body, and runner reports referenced by issue comments. It must not infer success from chat history.
- A review finding must include `summary`, `evidence`, `proposedFix`, `sourceIssue`, optional `sourcePr`, and `findingFingerprint`. The runner creates at most one `agent:manual` follow-up per fingerprint by searching GitHub for the fingerprint before creation.
- Review follow-up issues are titled `Self-improvement follow-up: <summary>`, labeled exactly `agent:manual` and `self-improvement`, and must never include `agent:auto`.

## 6. Execution Slices And Multi-Agent Model
- **Slices:** Slice 1: cleanup and local-only boundary, removing accidental package WIP and adding the ignored local path. Slice 2: shared preflight and atomic lock handling for repo identity, labels, auth, active lock, and stale lock recovery. Slice 3: local discovery command that runs the architecture discovery prompt, validates the JSON contract, fingerprints the first valid candidate, creates or reuses one `agent:auto` issue with `self-improvement`, and records state. Slice 4: targeted implementation command that runs `codex-orchestrator run --target . --issue <number>` for the created or reused discovery issue. Slice 5: post-implementation live smoke command that runs `npm run smoke:live`, records success/failure, and includes issue/PR references in the daily summary. Slice 6: local review command that scans eligible completed self-improvement issues, analyzes issue/PR/smoke evidence, and creates or reuses `agent:manual` follow-up issues for concrete workflow problems. Slice 7: local daily command that records separate phase results and always attempts review after discovery/implementation/live-smoke unless shared preflight failed. Slice 8: Codex App automation that runs the local daily command once per day.
- **Per-Slice Test/Proof:** Slice 1 proof: `git status --short` shows no package self-improvement files and `.gitignore` contains the local ignore rule. Slice 2 proof: lock acquisition is atomic and preflight fails before GitHub mutation when repo/labels/auth are wrong. Slice 3 proof: a controlled live run creates or reuses exactly one GitHub issue with `agent:auto`, `self-improvement`, ownership metadata, and remote idempotency markers. Slice 4 proof: targeted run uses the created/reused issue number and either opens a draft PR or records a runner blocker from codex-orchestrator. Slice 5 proof: after targeted implementation, `npm run smoke:live` runs once and its result is captured in local state and the daily summary. Slice 6 proof: review creates only `agent:manual` follow-up issues with finding fingerprints and never adds `agent:auto`. Slice 7 proof: injected implementation or live-smoke failure still allows review phase to run and summarize results. Slice 8 proof: Codex App automation is installed with a daily schedule and points at the local daily command.
- **Exit Gates:** Do not run `npm test` for ignored local-only script changes unless tracked package files change beyond `.gitignore`. Run `git status --short` to verify local runner files remain ignored. Run one live end-to-end daily command after approval because the user explicitly requested no dry-run. That live daily command must include post-implementation `npm run smoke:live`.
- **Agent Matrix:** Phase | Owner | Input | Output | Dependencies. Cleanup | main agent | current worktree | package WIP removed, ignore rule planned/applied | none. Preflight | local runner | config, gh auth, labels, lock | pass/fail shared gate | none. Discovery | local runner | repo docs/code and JSON skill output | one marker-matched `agent:auto` issue | preflight, `gh`, Codex. Implementation | codex-orchestrator run | created/reused issue number | draft PR or runner blocker | discovery issue. Live Smoke | local runner | post-implementation repo state | smoke pass/fail evidence | implementation phase completed. Review | local runner | self-improvement issue/PR/smoke evidence | marker-matched `agent:manual` follow-up issues | preflight. Automation | Codex App | local daily command | daily recurring run | local runner exists.
- **Parallelization Limits:** Discovery, targeted implementation, live smoke, and review run serially inside one daily lock. The local runner must not start a second `codex-orchestrator run` or `npm run smoke:live` while one self-improvement implementation/smoke sequence is active. Review follow-up creation must not run in parallel with another review pass over the same source issue. Remote idempotency markers remain mandatory even with the local lock.
